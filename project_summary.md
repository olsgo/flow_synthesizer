# Project Summary: Integrating Serum 2 with Flow-Synth

## 1. Central Purpose

The primary goal of this project is to successfully train the Flow-Synth model on synthesizers other than u-he Diva, which was used by the original developers. This will allow us to generate a wider variety of sounds and explore the capabilities of the Flow-Synth architecture with different synthesis engines.

Our first target is **Xfer Records Serum 2**, a powerful and popular wavetable synthesizer that we believe will yield amazing results when combined with Flow-Synth.

## 2. Environment Setup

All work on this project should be done within the **`flow-synth`** Conda environment.

## 3. The Challenge: Diva vs. Serum 2

The original Flow-Synth implementation relied on Diva's presets, which are stored in a human-readable format. This made it relatively straightforward to parse and manipulate them.

Serum 2, however, uses the proprietary `.serumpreset` format. This is a binary format that is not directly human-readable. Therefore, a different approach is required to extract the synthesis parameters.

## 4. Our Approach: The "Parameter Mapping" Method

To work with Serum 2, we have adopted a multi-step process inspired by the examples found in `/Users/gjb/Projects/flow_synthesizer/examples/`:

1.  **Decode `.serumpreset` files:** We use a Node.js utility (`node-serum2-preset-packager`) to convert the binary `.serumpreset` files into a human-readable JSON format. The converted presets are located in `/Users/gjb/Datasets/serum2/presets`, mirroring the original factory preset structure.

2.  **Load Serum 2 in DawDreamer:** We use the DawDreamer library to load the Serum 2 VST3 plugin.

3.  **Manual Parameter Mapping:** For each preset, we parse the corresponding JSON file and manually map the parameters found within it to the parameters exposed by the Serum 2 plugin in DawDreamer. This is the most critical and challenging part of the process.

**Important:** Directly loading `.vstpreset` files or presets in their original `.serumpreset` format will **NOT** work for our purposes. We must use the parameter mapping approach described above.

## 5. What We've Tried

We have made several attempts to implement the parameter mapping approach, with varying degrees of success:

- **Initial Parameter Mapping:** Our first attempts involved creating a direct mapping between the parameter names in the JSON files and the VST parameter names exposed by DawDreamer. This resulted in some audio being generated, but it was often glitchy and full of artifacts.

- **The "Hybrid" Approach:** We created a `serum2_hybrid_dataset_generator.py` script that combined our parameter mapping efforts with some of the original Flow-Synth code. This also produced glitchy audio, and we encountered numerous "Invalid value" warnings from DawDreamer, indicating that our parameter normalization was incorrect.

- **VST3 Preset Loading:** We experimented with loading `.vstpreset` files directly using DawDreamer's `load_vst3_preset` function. This failed because the `.serumpreset` files are not compatible with this function, and we do not have a reliable way to convert them to the `.vstpreset` format.

- **Debugging Glitchy Audio:** We created a `debug_glitchy_audio.py` script to isolate the source of the audio artifacts. This script confirmed that the Serum 2 plugin loads correctly in DawDreamer and can produce clean audio with its default settings. This strongly suggests that the glitches are caused by our parameter mapping and normalization logic.

## 6. Current Status: Stuck on Parameter Mapping and Normalization

We are currently stuck on the parameter mapping and normalization step. The audio generated by our scripts is still glitchy, and we are still seeing "Invalid value" warnings from DawDreamer.

The core of the problem seems to be that we do not have a complete and accurate understanding of how the parameter values in the JSON files correspond to the 0.0-1.0 floating-point values expected by DawDreamer.

## 8. Next Steps

The immediate next step is to perfect the parameter mapping and normalization. This will likely involve:

- A more detailed analysis of the parameter ranges and data types used by Serum 2.
- A more sophisticated normalization function that can handle the various types of parameter values found in the JSON files (e.g., percentages, decibels, enumerated values).
- Potentially creating a more comprehensive mapping dictionary that covers all of Serum 2's parameters.
